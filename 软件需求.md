1. 核心需求与规则梳理

为了确保开发时逻辑不混乱，我们先明确核心的“体力数学模型”：

* **最大体力值（$Max$）**：$200$
* **初始体力值**：$200$
* **恢复速率（$\Delta t$）**：每 $8$ 分钟回复 $1$ 点。
* **核心逻辑**：只要当前体力值 $Current < 200$，后台就要开始计算恢复。
* **关键机制（避免后台常驻省电）**：**绝对不要**在后台开一个每8分钟执行一次的定时器，这样会被安卓系统杀掉且极度耗电。正确的做法是“基于时间戳差值计算”。
> **公式：**
> 当用户打开App或小组件刷新时：
> 
> $$Current = \min(Max, \text{上次体力值} + \lfloor \frac{\text{当前时间} - \text{上次更新时间}}{8 \text{分钟}} \rfloor)$$
> 
>

4. 关键核心代码实现思路 (伪代码/关键逻辑)

4.1 体力恢复核心算法 (Kotlin)

每次用户打开App或点击小组件时，调用此函数刷新体力：

```kotlin
fun refreshStamina(context: Context) {
    val maxStamina = 200
    val intervalMillis = 8 * 60 * 1000 // 8分钟的毫秒数
    val currentTime = System.currentTimeMillis()

    // 1. 从存储中读取上次的数据
    val lastStamina = dataStore.getCurrentStamina()
    val lastTime = dataStore.getLastUpdateTime()

    if (lastStamina >= maxStamina) {
        // 如果原本就是满的，只需更新时间戳，体力保持200
        dataStore.saveData(maxStamina, currentTime)
        return
    }

    // 2. 计算时间差，算出应该恢复多少点
    val timeDiff = currentTime - lastTime
    val recoveredPoints = (timeDiff / intervalMillis).toInt()

    if (recoveredPoints > 0) {
        val newStamina = minOf(maxStamina, lastStamina + recoveredPoints)
        // 关键：新的时间戳不能直接设为当前时间，应该扣除“不满8分钟的余数”，保证计时精准
        val remainingMillis = timeDiff % intervalMillis
        val newSavedTime = currentTime - remainingMillis
        
        dataStore.saveData(newStamina, newSavedTime)
    }
}

```

4.2 体力扣除逻辑

当用户手动扣除 $X$ 点体力时：

```kotlin
fun consumeStamina(points: Int) {
    // 1. 先执行一次刷新，把过去一段时间自动回复的体力加上
    refreshStamina(context)
    
    // 2. 读取刷新后的体力
    val current = dataStore.getCurrentStamina()
    
    // 3. 扣除体力（最低为0）
    val finalStamina = maxOf(0, current - points)
    
    // 4. 保存，并把时间戳更新为当前时间（重新开始计算8分钟循环）
    dataStore.saveData(finalStamina, System.currentTimeMillis())
    
    // 5. 触发小组件刷新
    MyWidgetManager.update(context)
}

```

5. 开发注意事项与避坑指南 (Crucial Tips)

6. **图片权限问题（Android 11及以上重大坑）**：
如果用户从相册选了一张图作为背景，App重启后可能会因为**外部存储访问权限失效**导致背景变黑。
* **解决方案**：在拿到相册图片的 `Uri` 后，必须执行 `contentResolver.takePersistableUriPermission(uri, Intent.FLAG_GRANT_READ_URI_PERMISSION)`。这样App即使重启，也能永久读取这张图。


2. **小组件图片尺寸限制**：
安卓小组件的远程视图（RemoteViews）传递图片时有内存限制（不能超过几MB）。如果用户上传了一张 1080P 或 4K 的手机壁纸给小组件，直接加载会导致小组件崩溃。
* **解决方案**：加载小组件背景图时，必须使用 Coil 或 Glide 对图片进行 **Resize（裁剪/压缩）**，将其缩放到小组件实际显示的大小（例如 $500 \times 500$ 像素以内）。


3. **定时刷新小组件**：
虽然我们用了“时间戳差值算法”，但如果手机一直停在桌面，小组件也需要定时变动（比如每8分钟数字加1）。
* **解决方案**：在安卓的 `AndroidManifest.xml` 中为小组件配置 `updatePeriodMillis`（系统为了省电，最低限制为 30 分钟）。为了实现更精准的每8分钟刷新，可以使用 `WorkManager` 注册一个定期任务，在手机解锁或亮屏时顺便刷新小组件。
